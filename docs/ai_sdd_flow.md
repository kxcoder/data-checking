你提的这一点非常关键，也是从“功能实现”迈向“系统设计”的分水岭。当需求不再是某个具体的用户故事，而是要构建一个**可复用、可配置、可扩展的通用能力**时，你向 AI 传递信息的维度和格式都需要升级。

此时，单一的“用户用例”不足以描述需求，你需要提供的是**领域模型、可变性描述和约束规则**。OpenSpec + Superpowers 依然适用，但输入的“原料”变了。

### 🧠 核心思路转变

| 维度 | 具体功能开发 | 通用能力建设 |
| :--- | :--- | :--- |
| **输入重点** | 用户行为流程 | 系统抽象模型、可变点、配置规则 |
| **用例性质** | **场景化用例**（Who-Do-What） | **参数化用例**（What-Can-Vary & How） |
| **OpenSpec 设计重点** | 如何满足指定流程 | 如何定义稳定接口与扩展点 |
| **AI 需要的上下文** | 用户故事、原型图 | 领域词汇表、实体关系、可变性矩阵 |

### 📋 第一步：提供结构化“通用能力”上下文

在发出 OpenSpec 指令前，你需要先向 AI 提供一份“通用能力设计书”。这比用例列表更重要。可以用以下模板组织信息：

```markdown
## 1. 能力名称与目标
【例】通用积分规则引擎：允许业务方在不修改代码的情况下，定义积分获取和消耗的计算逻辑。

## 2. 领域模型（核心实体与关系）
- **积分事件 (PointEvent)**：描述一次积分变动的抽象记录，包含类型、来源、用户、数量、时间。
- **规则定义 (RuleDefinition)**：描述积分计算逻辑的元数据，包含条件表达式、计算公式、优先级。
- **积分账户 (PointAccount)**：用户的积分余额聚合根。

## 3. 可变性分析（什么需要做成可配置/可扩展的？）
| 变化点 | 变化维度 | 示例值 |
| :--- | :--- | :--- |
| 积分获取触发条件 | 业务动作 | 下单、评价、签到、分享 |
| 积分计算逻辑 | 公式 | `金额 * 0.1`、`固定值`、`连续签到天数 * 5` |
| 积分消耗校验 | 前置规则 | 最低兑换门槛、限品类、限时段 |
| 积分有效期 | 时间策略 | 固定日期、获取后N天、季度清零 |

## 4. 约束与不变性（什么必须始终保持？）
- 积分变动必须可追溯，日志不可篡改。
- 积分扣减必须遵循原子性，避免超扣。
- 规则变更不能影响已发生的历史流水。

## 5. 系统边界与接口草案
- 对外暴露：规则管理接口（CRUD）、积分查询接口、积分变动执行接口。
- 依赖上游：用户服务、订单服务（仅通过事件监听，避免强耦合）。
```

### 🎭 第二步：编写“参数化用例”而非“场景化用例”

对于通用能力，用例要从描述“具体用户做了什么”转向描述“系统如何响应不同的参数组合”。这类用例面向的是**系统管理员**或**集成开发者**。

**场景化用例（错误示例，用于具体功能）：**
> 家长在页面点击“增加积分”，选择孩子，输入5分，系统给该孩子加5分。

**参数化用例（正确示例，用于通用能力）：**
> ### 用例：管理员配置一条新的积分获取规则
> - **参与者**：运营管理员
> - **前置条件**：积分规则引擎已部署，规则元数据模型已定义。
> - **主流程**：
    >   1. 管理员进入规则配置页面。
>   2. 系统展示当前生效的规则列表及优先级。
>   3. 管理员选择“新建规则”，并指定触发事件类型（如：`order.paid`）。
>   4. 管理员在条件编辑器中输入：`order.amount >= 100`。
>   5. 管理员在公式编辑器中输入：`floor(order.amount / 10)`。
>   6. 管理员设置规则有效期为永久，优先级为10。
>   7. 管理员提交，系统校验条件表达式语法和公式返回值类型，校验通过后持久化规则。
> - **后置条件**：新规则生效，后续匹配的订单事件将按此规则计算积分。

> ### 用例：积分规则引擎响应订单支付事件
> - **参与者**：消息队列监听器（系统组件）
> - **前置条件**：接收到 `order.paid` 事件，事件体中包含 `userId`、`orderId`、`amount`。
> - **主流程**：
    >   1. 引擎加载所有状态为“启用”的积分获取规则，按优先级排序。
>   2. 遍历规则，使用事件数据作为上下文，评估条件表达式。
>   3. 若条件为真，则使用公式计算积分数值 `earnedPoints`。
>   4. 引擎调用积分账户服务，原子增加该用户的积分，并记录一条类型为“订单奖励”的流水，流水扩展属性中保存 `orderId`。
>   5. 终止遍历（若策略为“首个匹配”）。
> - **异常流程**：
    >   - 2a. 若表达式语法错误，记录错误日志并跳过该规则。
>   - 4a. 若积分账户服务不可用，则重试或进入死信队列。

### 🔗 第三步：在 OpenSpec 中描述通用能力的“规范”

对应完整的提示词：

```markdown

请使用 OpenSpec 为【通用积分规则引擎】创建一个变更提案。

## 1. 核心用例（功能行为基线）
以下用例描述了系统管理员与规则引擎的交互行为，以及引擎对外部事件的响应方式。设计必须覆盖这些用例。

### 用例 UC-01：管理员配置积分获取规则
- **参与者**：运营管理员
- **前置条件**：管理员已登录后台系统
- **主流程**：
  1. 管理员进入规则配置页面，系统展示当前生效规则列表（按优先级排序）。
  2. 管理员点击“新建规则”，并选择事件类型为 `order.paid`。
  3. 管理员在条件编辑器中输入：`context.order.amount >= 100`。
  4. 管理员在公式编辑器中输入：`context.order.amount * 0.1`。
  5. 管理员设置优先级为 10，有效期为永久，然后提交。
  6. 系统验证表达式语法、函数白名单合规、事件类型存在，验证通过后保存规则。
  7. 系统返回规则列表，新规则出现在正确优先级位置。
- **异常流程**：
  - 6a. 表达式语法错误，系统返回错误信息并拒绝保存。
  - 6b. 公式返回值非数字类型，系统提示“公式必须返回数值”。

### 用例 UC-02：订单支付事件触发积分发放
- **参与者**：消息队列监听器（系统组件）
- **前置条件**：接收到 `order.paid` 事件，事件包含 `userId`、`orderId`、`amount`。
- **主流程**：
  1. 引擎加载事件类型为 `order.paid` 且状态为启用的规则，按优先级排序。
  2. 遍历规则，将事件数据作为上下文，评估条件表达式。
  3. 当首个条件为真的规则被匹配，使用公式计算积分数额（向上取整）。
  4. 引擎原子增加用户积分余额，记录流水，流水扩展属性存储 `orderId`。
  5. 引擎返回成功，结束遍历。
- **异常流程**：
  - 4a. 同一 `orderId` 已处理过（幂等性校验），引擎直接返回成功，不重复发放。
  - 4b. 积分账户并发冲突，引擎重试最多 3 次，失败则进入死信队列。

### 用例 UC-03：积分过期策略配置与执行
- **参与者**：定时任务调度器（系统组件）
- **前置条件**：系统存在多种积分有效期策略（固定日期、获取后 N 天、季度清零）。
- **主流程**：
  1. 定时任务每日凌晨触发过期扫描。
  2. 引擎加载所有启用的有效期策略，遍历用户积分流水（只追加日志）。
  3. 根据策略计算每笔未过期积分的过期时间，若已过期则标记。
  4. 引擎扣减对应积分总额，记录过期扣减流水。
- **扩展要求**：允许通过实现 `PointExpiryPolicy` 接口注册新策略。

## 2. 领域模型（数据抽象）
- 积分事件 (PointEvent)：记录积分变动的抽象实体，属性包含 eventId, userId, changeAmount, eventType, sourceId, timestamp, metadata(扩展属性)。
- 规则定义 (RuleDefinition)：定义计算逻辑的元数据，属性包含 ruleId, name, eventType, conditionExpression, formulaExpression, priority, status, validFrom, validTo。
- 积分账户 (PointAccount)：聚合根，属性包含 accountId, userId, totalPoints, version。

## 3. 可变性分析（扩展点设计）
| 变化点 | 配置方式 | 技术要求 |
| :--- | :--- | :--- |
| 触发事件类型 | 枚举值 | 支持动态注册新的事件类型，不修改核心代码 |
| 条件表达式 | 字符串表达式 | 使用安全的表达式引擎（如 SpEL 或 MVEL），支持沙箱限制 |
| 积分计算公式 | 字符串表达式 | 同上，返回值必须是数字类型 |
| 规则优先级 | 整型字段 | 支持排序，同事件类型按优先级从高到低匹配 |
| 积分有效期策略 | 策略模式扩展点 | 预留接口 PointExpiryPolicy，支持固定日期、获取后N天等多种实现 |

## 4. 约束与不变性
- 积分流水一旦生成，不可修改或删除，必须使用只追加日志模式。
- 积分扣减操作必须原子化，使用乐观锁或数据库行锁避免超扣。
- 同一事件（相同 sourceId）不能重复发放积分，需实现幂等性校验。
- 规则表达式仅允许访问上下文对象 `context` 的属性和一组白名单函数（如 `max()`, `min()`, `now()`）。

## OpenSpec 输出要求
请基于以上信息，在项目中创建标准的 OpenSpec 变更目录，并生成以下文件：
1. `proposal.md`：说明为何需要构建此通用能力，以及它将如何影响系统架构。
2. `design.md`：详细阐述技术方案，必须包括：
    - 表达式引擎的选型对比（至少对比两种方案并给出推荐）。
    - 扩展点的设计（类图或接口描述）。
    - 幂等性实现方案（如基于 sourceId + eventType 的唯一索引设计）。
    - 并发扣减方案（乐观锁 vs 悲观锁的权衡）。
3. `specs/point-engine/spec.md`：定义能力规格，必须包含：
    - 规则配置接口的 JSON Schema 定义。
    - 积分变动接口的请求/响应格式和幂等性头要求。
    - 表达式语法的约束说明（白名单函数、禁止的操作）。
4. `tasks.md`：将实现过程拆解为可验证的任务清单，按“核心模型 → 表达式引擎集成 → 规则执行器 → 对外接口”的顺序排列。

请确保所有文档使用清晰、专业的中文撰写，代码示例可使用英文。



```


有了上述上下文，OpenSpec 生成的 `specs/` 文档内容会更加抽象和强约束：

*   `specs/point-rule-engine/spec.md` 会包含：
    *   规则数据结构的 JSON Schema 定义。
    *   表达式引擎的函数白名单和安全沙箱要求。
    *   积分变动的**幂等性**要求（同一订单不能重复发放积分）。
*   `design.md` 会详细说明：
    *   为何选择某种表达式语言（如 MVEL、SpEL 还是自研 AST）。
    *   如何通过事件溯源保证积分流水可追溯。

### 🤖 第四步：Superpowers 如何适应通用能力开发？

Superpowers 在执行通用能力开发时，其 TDD 策略也会调整：

*   **单元测试焦点**：不再测试“用户点击按钮后页面跳转”，而是测试：
    *   `RuleEvaluator.evaluateCondition()` 在不同输入下的正确性。
    *   `PointCalculator.calculate()` 在公式 `amount * 0.1` 下的精度。
    *   `PointService.addPoint()` 的并发安全性和幂等性。
*   **集成测试焦点**：模拟 MQ 消息，验证端到端的规则匹配与积分发放流程。

### 💎 总结：向 AI 交付一份“技术设计任务书”

当你要求 AI 构建一个通用能力时，**你不能只给它用例，而必须给它一份浓缩了你领域知识的“技术设计任务书”**。这份任务书包含：

1.  **领域模型图/描述**（AI 据此建立数据模型）。
2.  **可变性表格**（AI 据此设计配置项和扩展点）。
3.  **约束清单**（AI 据此生成防御性代码和校验逻辑）。

**最终指令示例**：
> “请基于我提供的**通用积分规则引擎设计上下文**（包含领域模型和可变性分析），使用 OpenSpec 为该能力创建一个变更提案。设计目标是为整个系统提供可配置、可扩展的积分计算与扣减基础服务。请生成完整的提案文档，并在 `design.md` 中重点阐述扩展点设计和表达式引擎选型考量。”

通过这种方式，OpenSpec 生成的蓝图不再局限于某个具体页面，而是一个**可支撑未来多种业务场景的健壮骨架**。Superpowers 随后会以极高的代码质量将这个骨架实现出来。
