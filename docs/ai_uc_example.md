# 面向 AI 的后端技术用例模板
你需要将产品需求分析后，整理成一份结构化的后端设计上下文。技术用例列表是其中的核心部分。

## 1. 服务边界与职责
- **本服务名称**：积分规则引擎服务 (point-engine)
- **核心职责**：接收业务事件，根据可配置的规则计算并执行积分变动，保证幂等性和一致性。
- **对外提供**：规则管理接口（内部）、积分变动执行接口（内部）、积分查询接口。
- **依赖上游**：订单服务（通过 MQ 推送事件）、用户服务（校验用户合法性）。
- **不负责**：前端页面渲染、用户权限校验（由网关/上游负责）。

## 2. 技术用例列表（核心）

### 用例 SYS-UC-01：处理业务事件并触发积分计算
- **参与者**：消息队列监听器（系统）
- **触发条件**：接收到规范的业务事件消息（如 `order.paid`）。
- **前置条件**：消息格式符合约定 Schema，携带必要的业务标识（`sourceId`, `userId`）。
- **主成功流程**：
    1. 解析消息，校验签名与时效性。
    2. 基于 `sourceId` 进行幂等性检查，若已处理则直接返回 ACK。
    3. 加载该事件类型下所有启用状态的规则，按优先级排序。
    4. 遍历规则，使用安全表达式引擎评估 `conditionExpression`。
    5. 命中断首个满足条件的规则后，执行 `formulaExpression` 计算积分值。
    6. 调用账户服务原子增加积分，同时写入一条不可变流水记录（关联 `sourceId`）。
    7. 返回 ACK 确认消费。
- **异常与补偿流程**：
    - 4a. 表达式评估异常：记录错误日志，跳过该规则继续评估下一条（或终止并进入死信）。
    - 6a. 账户服务调用失败（超时/宕机）：不确认消息，触发 MQ 重试机制。
    - 6b. 并发导致账户余额版本冲突：乐观锁重试 3 次，失败则转异步补偿任务。
- **技术约束**：
    - 处理必须是**幂等**的（通过 `sourceId` 唯一键保证）。
    - 规则评估必须在**沙箱**内执行，禁止访问系统资源。

### 用例 SYS-UC-02：管理员配置积分规则
- **参与者**：内部管理后台（或 API 调用方）
- **主成功流程**：
    1. 接收 HTTP POST 请求，包含规则定义 JSON。
    2. 校验请求头中的认证 Token（由网关透传，服务仅验证格式）。
    3. 使用预定义的语法校验器检查 `conditionExpression` 和 `formulaExpression` 的**安全性**和**语法正确性**。
    4. 持久化规则至数据库，若优先级冲突则按时间戳自动调整。
    5. 返回规则 ID 及创建时间。
- **异常流程**：
    - 3a. 表达式包含非法函数调用（如 `System.exit()`）：返回 400 错误并指出具体非法符号。
    - 4a. 数据库写入冲突：返回 409 冲突错误，提示客户端刷新后重试。
- **扩展点**：表达式语法校验器需支持**插件化扩展**，以便未来升级表达式引擎。

### 用例 SYS-UC-03：积分过期扫描与扣减
- **参与者**：定时任务调度器（Cron）
- **主成功流程**：
    1. 任务触发，获取分布式锁（保证单实例执行）。
    2. 分页查询未过期的积分流水记录（状态为 `AVAILABLE`）。
    3. 逐条加载对应的过期策略 Bean（如 `FixedDatePolicy`, `RelativeDaysPolicy`）。
    4. 策略计算 `expiryTime`，若已小于当前时间，则执行原子扣减。
    5. 记录扣减流水，标记原流水状态为 `EXPIRED`。
- **非功能约束**：
    - 扫描过程必须支持**断点续传**，避免长事务。
    - 每次扣减必须记录 `traceId`，便于审计。
